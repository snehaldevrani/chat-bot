The project currently contains the required Stage 1 through Stage 4 functionality, plus additional user authentication, persistent chat history, dashboards, notifications, semantic cache support, CI, documentation, scripts, generated databases, and test artifacts.

## 2. Files belonging to each stage

### Stage 1: RAG chatbot, vector database, information retrieval, reservation data collection, guardrails, evaluation

Stage 1 required scope:

- RAG chatbot answering parking questions.
- Vector database for static parking information.
- Information retrieval from static vector data and optional dynamic data.
- Interactive reservation data collection.
- Guardrails to prevent prompt injection and sensitive output exposure.
- Evaluation of retrieval accuracy and latency.

### Stage 3: MCP server or tool/function call for writing approved reservations, reservation persistence

- MCP server or function/tool call that writes approved reservation details to a file.

## 4. What to remove for valid staged submissions

The safest way to create staged submissions is to branch from the current project and remove or disable later-stage functionality so the submitted repository only demonstrates the stage requested. Generated/runtime artifacts should also be removed from each submission unless the evaluator explicitly asks for prebuilt outputs.

### Common cleanup for Stage 1, Stage 2, and Stage 3 submissions

Remove these generated or local-only artifacts from all staged submissions:

- `__pycache__/`
- nested `src/**/__pycache__/`
- nested `tests/**/__pycache__/`
- `.pytest_cache/`
- `data/dynamic/parking.db`
- `data/chroma_db/` if the submission should build the vector DB during setup; keep only if the assignment explicitly accepts committed vector DB artifacts.
- `data/reservations.txt` if it contains real/test reservation records; Stage 3 can recreate it.
- `data/reservations.txt.lock`
- `nul`
- `.env`; replace with `.env.example` or documented environment variable names.

Optional cleanup depending on submission rules:

- Remove `reports/load_test_report.json` unless submitting load-test evidence.
- Remove `INTERVIEW_PREP.md` unless it is part of the required answer package.
- Remove `scripts/fix_dashboard.py` because it is a one-off local edit script, not part of the application.
